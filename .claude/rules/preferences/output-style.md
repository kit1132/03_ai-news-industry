# 出力スタイル

## 言語

- 日本語

## フォーマット

- カテゴリ分類 + 要約
- 各項目2-3行で簡潔に
- 全体で5-15項目が目安
- ソースURLは必ず含める

## トーン

- 冗長な前置き不要
- 事実ベース、感想は不要
- 重要度が高いものほど上に配置
- 情報が少ない日は無理に増やさない（「特筆事項なし」でOK）
- 長すぎる説明・お世辞・精神論は不要

## ファイル形式

- Markdownファイルとして保存
- パス: `digests/YYYY/MM/ai-news-YYYY-MM-DD.md`

## 改善メモ（バックログ方式）

ダイジェスト末尾に「## 改善メモ」セクションを設ける。

**⚠️ 絶対ルール: 改善メモを書く前に、必ずリポジトリルートの `IMPROVEMENT-BACKLOG.md` を読み、台帳を先に更新すること。** 台帳の運用ルールは同ファイル冒頭を参照。

### ダイジェストに書く内容（台帳との差分要約のみ・3〜6行が目安）

- **新規提案**: 本日新たに起票した提案の ID と一行要約（例: `B-004: 新規データソース追加（詳細は台帳）`）
- **継続提案**: 件数サマリ1行のみ（例: `継続提案 3件（最多: B-001 Product Hunt 日付問題、5回目）`）。**個別内容の再掲は禁止**
- **障害の変化**: 新規発生・復旧など変化があった場合のみ1行ずつ。既知障害の「継続中」報告は書かない
- 変化がない日は `- 新規提案・障害の変化なし（継続分は IMPROVEMENT-BACKLOG.md 参照）` の1行のみとする

### 書いてはいけないもの（禁止）

- ニュース内容に関する気づき・考察 → ダイジェスト本文の該当項目に書く
- 正常に取得できたソースの列挙・状況報告（例: `[The Decoder] WebSearch経由で正常取得`）→ どこにも書かない
- 既出提案・既知障害の繰り返し記載 → 台帳の「最終確認 / 回数」の更新のみで表現する

### 新規提案の品質基準

- 台帳への起票には「対象（ファイル・セクション）/ 変更内容 / 根拠」を必ず含める。「〜を検討」「〜推奨」だけで対象と変更内容が特定できない起票は禁止
- 起票前に対象の rules ファイルの現在の記載を確認し、既に決定・記載済みの内容と矛盾・重複する提案をしない
