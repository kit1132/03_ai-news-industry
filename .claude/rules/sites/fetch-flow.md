# ソース取得フロー

## 取得手順

各ソースの取得は以下の順序で試行する。

1. **プライマリURLをWebFetchで取得**
2. 403 / 429 / タイムアウトの場合 → **代替URLがあればWebFetchで試行**
3. それでも失敗した場合 → **WebSearchで `site:ドメイン 検索キーワード 当年` を実行**
4. `site:` 検索で結果0件の場合 → **`site:` を外して `検索キーワード 当年` で再検索**
5. すべて失敗した場合 → ダイジェストに「取得不可（手動確認推奨）」と記載

## フォールバック発生時の記録ルール

フォールバック（手順2以降）を使用した場合、リポジトリルートの **`IMPROVEMENT-BACKLOG.md`「既知の取得障害」セクションを更新する**。ダイジェストの改善メモに書くのは「変化があったもの」だけ。

- **新規の障害**（台帳にない URL）: 台帳に以下の形式で1行追加し、ダイジェストの改善メモにも「障害の変化」として1行記載する

  ```
  - <ソース名 / URL>: <403/429/timeout>（初出 YYYY-MM-DD / 最終確認 YYYY-MM-DD）→ 回避策: <成功した方法 / なし（取得不可・手動確認推奨）>
  ```

- **既知の障害**（台帳にある URL）: 台帳の「最終確認」日付を更新するのみ。**ダイジェストには書かない**
- **復旧**（台帳で障害扱いの URL が正常取得できた）: 台帳の該当行の行末に `→ 復旧（YYYY-MM-DD）` を追記し、ダイジェストの改善メモに「障害の変化」として記載する。行は削除しない
- **回避策が安定している長期障害**: `daily-sources.md` の取得方法欄を回避策ベースに書き換える提案を、台帳の「提案中」に起票する（既に同趣旨の提案があれば回数を +1 するだけ）

## WebSearch利用時の注意事項

- `site:` オペレータは非対応の場合がある。結果0件なら必ず `site:` を外して再検索する
- WebSearchに日付フィルタ機能はない。クエリに年（例: `2026`）を含めて新しい結果を優先する
- インデックス反映に数時間〜数日のラグがある。速報性が必要な情報には限界がある
- WebSearch結果はURLとスニペットのみの場合がある。必要に応じて結果内のURLをWebFetchで追加取得する
